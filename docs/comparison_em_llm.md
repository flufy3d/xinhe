# EM-LLM vs 心核 (Xinhe) — 架构对比

> EM-LLM: https://github.com/em-llm/EM-LLM-model (ICLR 2025)
> 论文: https://openreview.net/forum?id=BI2int5SAC

两个项目都在解决 **LLM 长期记忆** 问题，但思路截然不同。

---

## 一句话总结

- **EM-LLM** = 外存 + 检索：把 KV cache 卸载到 CPU/磁盘，需要时用惊讶度分割 + 相似度检索找回来。
- **心核** = 内存 + 演化：用可学习的 state token 持续压缩和演化记忆，记忆是模型状态的一部分。

---

## 1. 根本哲学

| | EM-LLM | 心核 |
|---|---|---|
| **核心思路** | 外挂检索系统，从历史 KV cache 中找回相关片段 | 内生状态演化，让模型学会自己"记住"什么 |
| **类比** | 人翻笔记本找信息 | 人脑把经验内化为直觉/记忆 |
| **记忆形态** | 显式存储的 KV 对（episodic memory blocks） | 32 个可学习 state token 的持续演化 |

## 2. 训练 vs 推理

| | EM-LLM | 心核 |
|---|---|---|
| **是否需要训练** | 完全免训练（inference-time monkey-patch） | 需要训练（LoRA + StatePlugin，~500K 参数） |
| **修改权重** | 不动模型权重，运行时替换 attention forward | 冻结 backbone，只训练 state 机制和 LoRA |
| **在线学习** | 无 | 核心设计 — state 随对话持续演化 |
| **持续适应** | 无 — 模型永远是出厂状态 | sleep 时反向传播更新权重，对话经验固化进 .pt |

## 3. 记忆机制

| | EM-LLM | 心核 |
|---|---|---|
| **存储方式** | KV block 卸载到 CPU/磁盘，按需检索回 GPU | state token 始终在 GPU 上，参与每次 forward |
| **记忆容量** | 理论无限（已测到 1000 万 token） | 受 state 维度限制（32×768），是压缩瓶颈 |
| **分割策略** | 惊讶度阈值 + 图论社区检测（modularity 优化） | 无显式分割 — gate 机制隐式决定保留/遗忘 |
| **检索策略** | 两阶段：相似度 top-k + 时间邻近 contiguity buffer | 无检索 — state 本身就是压缩后的记忆 |
| **遗忘机制** | 无主动遗忘，所有 block 永久存储 | dual-layer gate 持续决定更新/保留 |
| **记忆层级** | 单层（KV block 存储） | 双层 — state 为短期记忆，sleep 权重更新为长期记忆 |
| **跨会话持久** | 无 — 会话结束 KV 清空 | .pt 文件保存全部状态和权重，重启后灵魂延续 |

## 4. 技术细节对比

### EM-LLM 的核心流程

```
输入 token 流
    ↓
按 chunk (512 token) 处理
    ↓
计算每个 token 的惊讶度 (-log P(x_t))
    ↓
超过阈值 (γ·std + mean) 的位置 → 事件边界候选
    ↓
图论 modularity 优化 → 精化边界位置
    ↓
KV 对按事件分块存储 → CPU/磁盘
    ↓
解码时: initial tokens (128) + 检索 tokens (2048) + local window (4096)
    ↓
两阶段检索: 代表向量相似度 top-k + 时间邻近 buffer
```

**有效窗口**: ~6272 tokens（固定，不随总长度增长）

### 心核的核心流程

```
输入 segment (256 token)
    ↓
state token (32×768) 注入到序列前端
    ↓
冻结 backbone + LoRA 前向传播
    ↓
提取输出中的 state 部分
    ↓
dual-layer gate: gate = σ(bias + proj([old, new]))
    ↓
state_next = gate · state_old + (1-gate) · state_new
    ↓
每 4 个 segment → sleep 巩固 (state-only self-attention)
    ↓
state 持续流入下一个 segment

    ═══ sleep 权重更新 ═══
    ↓
对话积累足够经验后触发 sleep
    ↓
state 信息通过反向传播固化进 LoRA + StatePlugin 权重
    ↓
.pt 保存 → 下次加载时，权重已包含历史经验
```

**有效记忆**: 双层 — state token 为工作记忆（短期），权重为固化记忆（长期）

## 5. 生物学灵感

| | EM-LLM | 心核 |
|---|---|---|
| **认知理论** | Event Segmentation Theory（事件分割理论） | 多时间尺度记忆 + 睡眠巩固 |
| **对应机制** | 惊讶度 → 事件边界，时间邻近 → 回忆扩散 | gate bias → 脑区特化，sleep → 记忆整合 |
| **时间尺度** | 单一（block 粒度） | 多尺度涌现（gate bias 自动分化快/慢区域） |
| **认知验证** | 事件边界与人类感知强相关（论文已验证） | 待验证 — 观察 gate 分布是否涌现时间尺度分离 |

## 6. 适用场景

| | EM-LLM | 心核 |
|---|---|---|
| **擅长** | 超长文档（书籍、代码库）、精确事实检索 | 多轮对话中的渐进式理解、人格/偏好积累、个性化 AI |
| **模型规模** | 7B-8B（Mistral, LLaMA-3, Phi-3） | MiniMind (~180M) 验证 → Qwen-7B 迁移 |
| **上下文规模** | 百万级 token（已测 10M） | 单次 episode 千级 token，但 state + 权重可跨会话无限积累 |
| **信息保真度** | 高 — 原始 KV 对完整保留 | 有损压缩，但 sleep 权重更新可将关键信息固化为模型能力 |
| **计算开销** | 内存随历史线性增长（CPU/磁盘） | 固定开销（32 token 常驻 GPU） |
| **个性化** | 无 — 所有用户共享同一模型 | 每个用户独立 .pt，经 sleep 分化成不同"灵魂" |

## 7. 核心取舍

### EM-LLM 的优势

- **即插即用**: 不需要训练，直接用在任何支持的 HuggingFace 模型上
- **无损记忆**: 原始 KV 完整保留，检索到就是精确信息
- **已验证规模**: 在 LongBench 上超越 RAG 和全上下文模型，10M token 级别可用
- **工程成熟**: 有完整的 GPU/CPU/磁盘三级缓存体系

### EM-LLM 的局限

- **模型没有"学到"记忆**: 只是工程上的 KV 管理，模型本身对记忆机制一无所知
- **检索依赖**: 如果代表向量没有捕捉到语义，就检索不到
- **无抽象能力**: 不能从多次经验中提炼出高层模式
- **无遗忘**: 所有信息永久保存，没有主动的信息筛选

### 心核的优势

- **真正的学习**: state 通过梯度学会什么值得记、什么该忘
- **压缩抽象**: 有限 state 空间迫使模型提炼信息，可能涌现高层表示
- **多时间尺度**: gate bias 可以自组织出快变量/慢变量的分化
- **睡眠巩固**: 无输入时的自组织，离线知识整合
- **双层记忆**: state = 工作记忆（快，随对话变），权重 = 长期记忆（慢，sleep 时固化）。这对应人类海马体 → 皮层的记忆巩固过程，EM-LLM 没有这个层级
- **灵魂分化**: 同一个预训练起点，不同用户的 .pt 经 sleep 后走向不同的人格/知识结构。EM-LLM 的模型对所有用户完全相同
- **跨会话连续性**: .pt 保存即灵魂保存，重启后延续。EM-LLM 每次会话从零开始

### 心核的局限

- **单次容量瓶颈**: 32 个 state token 的即时工作记忆有限（但 sleep 权重更新缓解了这个问题 — 重要信息最终固化进权重，不需要 state 一直"端着"）
- **精确检索弱**: 压缩本质决定了不适合"第 372 页第 3 段说了什么"这类任务（State-level RAG 可补足，见下文）
- **规模待验证**: MiniMind 上验证后需迁移到 Qwen-7B 确认可扩展性

## 8. 互补关系

两者本质上是互补的：

```
EM-LLM  →  "图书馆"  →  海量信息的精确存取
心核     →  "大脑"    →  经验的内化与抽象
```

EM-LLM 解决的是 **"如何在海量历史中精确找到需要的信息"**，
心核解决的是 **"如何让模型真正内化经验、形成持久理解"**。

一个是图书馆，一个是大脑。理想的系统两者兼备。

### 具体融合路径：State-level RAG

心核 roadmap 中的 **State-level RAG** 是两者思路的直接结合：

```
传统 EM-LLM:  存储 KV block → 检索 KV block → 拼回 attention
State-level RAG:  存储历史 state 快照 → 检索相关 state → 融合进当前 state
```

不检索原始 token，而是检索**已经压缩过的状态**。这意味着：
- 检索单元是"理解"而不是"文本"，语义密度远高于 KV block
- 融合方式是 state 空间内的混合，而不是拼接进 attention window
- 存储开销极小（每个快照 32×768 float = ~100KB vs EM-LLM 的完整 KV）

### 心核独有的能力维度

以下能力在 EM-LLM 的框架中**没有对应概念**：

| 能力 | 机制 | 意义 |
|------|------|------|
| **灵魂分化** | 同一起点 → 不同用户 sleep → 不同 .pt | AI 个性化不是 prompt 工程，而是权重级别的分化 |
| **灵魂遗传** | 从成熟 .pt 初始化新 AI | 类似"师徒"关系，知识和风格可以传承 |
| **多 AI 社交** | 不同 .pt 之间交换 state | AI 之间的"交流"不需要自然语言，直接在状态空间对话 |
| **权重级记忆** | sleep 反向传播更新 LoRA | 反复出现的模式从 state 沉淀为权重，类似肌肉记忆 |

这些能力共同指向一个 EM-LLM 没有触及的问题：**AI 不只是处理信息，而是在积累经验中成长**。
